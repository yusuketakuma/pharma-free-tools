## 2. 戦略候補の評価マトリクス

| 戦略 | 期待リターン | リスク | 技術難易度 | 資金要件 | 優先度 |
|------|------------|-------|----------|---------|--------|
| **Event-driven（結果裁定）** | 高 | 中 | 低-中 | 低 | ★★★★★ |

## 3. 各戦略の具体的手法

### 3.1 Event-driven（結果裁定）

**概要**: 確定イベント（スポーツ結果、選挙結果等）の公表情報と市場価格の乖離を利用。

**具体的手法**:
1. **スポーツ結果監視**
   - 公式データソース（API提供者: SportsData.io, ESPN API等）をpolled
   - 試合終了直後、市場価格が解決前に0/1に収束するタイムラグを狙う
   - 例: 試合が終了し勝者が確定しているのに、市場価格が$0.85-0.90で推移

2. **情報源の優位性設計**
   - 複数ソースのクロスチェック（最低2ソースで一致）
   - WebSocket RTDS（Real-Time Data Service）の活用
   - Polymarket自身のresolution遅延を利用するタイムウィンドウ

3. **執行要件**
   - latency: 数秒以内の執行が理想
   - 注文タイプ: FOK（即時約定、一部約定なし）を基本
   - サイズ: 1回あたり最大資金の5%まで

### 3.2 Mispricing検出（情報優位性）

**概要**: 市場参加者の非効率性を利用したアービトラージ的アプローチ。

**具体的手法**:
1. **相互排他市場の裁定**
   - Yes/Noペアの合計が$1から乖離している場合にアービトラージ
   - 複数関連市場（Aが起きればBは起きない等）の論理的矛盾を検出

2. **ベイジアン推定による価格評価**
   - 現在の情報を基にした確率推定値と市場価格の差分を算出
   - 誤差が閾値を超えた場合にエントリー

3. **外部データとの比較**
   - 選挙: ポーリングデータの集計（FiveThirtyEight, RealClearPolitics等）
   - スポーツ: オッズコンパレータ（Pinnacle, Betfair等）との比較
   - 経済指標: 公式発表値と予測市場の乖離

## 4. リスク管理戦略

### 4.1 資金管理

```
基本ルール:
- 初期資金: $100（損失可能額）
- 1トレード最大: 総資金の5%（$5）
- 1市場最大エクスポージャー: 総資金の20%（$20）
- 日次損失上限: 総資金の10%（$10）→到達で当日停止
- 週次損失上限: 総資金の20%（$20）→到達で1週間停止
- 利益確定: 月次+30%で資金の50%を引き出し
```

### 4.2 市場リスク

| リスク | 内容 | 緩和策 |
|--------|------|--------|
| **Resolution risk** | 不正解決、遅延、オーバーターン | 高確度イベントのみ。resolution sourceを事前確認。neg risk市場の挙動理解 |
| **Liquidity risk** | 大量注文で価格悪化 | 薄い市場は回避。スプレッドが広い市場はスキップ |
| **Counterparty risk** | 取引所のトラブル | 分散投資しない（Polymarket単一）。資金は最小限 |
| **Inventory risk** | 持ち越し中の価格変動 | 短期決済志向。イベント終了まで持ち越さない |

### 4.3 技術リスク

| リスク | 内容 | 緩和策 |
|--------|------|--------|
| **API障害** | レート制限・障害 | exponential backoff。複数endpoint fallback |
| **Latency** | 競合BOTとの速度差 | サーバー配置最適化。WebSocket優先 |
| **秘密鍵漏洩** | 資金喪失 | HSM or env vault。鍵は.envに置かず暗号化。CIに含めない |
| **バグ執行** | 誤注文 | kill switch。金額上限ハードコード。dry-runモード常時併用 |
| **Rate limit** | 25 req/min (relayer) | 注文頻度の制御ロジック。キューイング |

### 4.4 法規制リスク

| リスク | 内容 | 緩和策 |
|--------|------|--------|
| **日本居住者の取引可否** | geoblock / ToS | **Phase 0で必ず確認**。blockedに含まれる場合はVPN等の利用がToS違反の可能性 |
| **暗号資産税** | 申告義務 | すべての取引記録を保存。雑所得 or 事業所得の区分は税理士確認 |
| **賭博法** | 予測市場の法的位置づけ | 日本での法的グレーゾーン。専門家確認推奨 |

## 5. 監視・運用体制

### 5.1 必須KPI（ダッシュボード）

```
リアルタイム:
- 現在の資産額
- オープンポジション一覧
- 当日のPnL
- API response time
- Error rate

日次:
- 日次PnL
- トレード数 / 勝率
- 平均スプレッド / スリッページ

週次:
- 週次PnL / Sharpe ratio
- 最大ドローダウン
- 戦略別パフォーマンス
- レートリミットヒット回数
```

### 5.2 アラート設定

```
CRITICAL（即時対応）:
- 日次損失上限到達
- API認証エラー
- 秘密鍵関連エラー
- 予期しない大量注文

WARNING（1時間以内確認）:
- レートリミット接近（80%）
- 連続敗北（5回以上）
- 市場流動性急減
- WebSocket切断

INFO（日次確認）:
- 新規高ボリューム市場
- 異常価格変動
- システムリソース状況
```

### 5.3 Kill Switch

```
実装要件:
1. 手動: Telegram等からのコマンドで即時停止
2. 自動: 日次損失上限到達時
3. 自動: 連続N回のエラー時
4. ハードコード: 1注文上限金額（絶対超過不可）
5. ハードコード: 1日最大注文数（絶対超過不可）
```

## 6. 技術アーキテクチャ概要

```
┌─────────────────────────────────────────────────┐
│                  監視・ダッシュボード              │
│         (Grafana / 簡易Web / Telegram)           │
└──────────────────┬──────────────────────────────┘
                   │
┌──────────────────▼──────────────────────────────┐
│              オーケストレータ                     │
│   - スケジュール管理                               │
│   - kill switch / アラート判定                     │
│   - 戦略切り替え                                   │
└──┬──────────┬──────────┬──────────┬────────────┘
   │          │          │          │
┌──▼──┐  ┌───▼───┐  ┌──▼──┐  ┌───▼────┐
│データ│  │シグナル│  │執行  │  │リスク   │
│取得  │  │生成   │  │エンジン│  │マネージャ│
│層   │  │層    │  │     │  │        │
└──┬──┘  └───┬───┘  └──┬──┘  └───┬────┘
   │         │         │         │
   │    ┌────▼─────────▼─────────▼────┐
   │    │        共通DB / キャッシュ     │
   │    │   (SQLite / Redis / JSONL)    │
   │    └──────────────────────────────┘
   │
┌──▼──────────────────────────────────────────┐
│            Polymarket CLOB API               │
│  (Gamma API / WebSocket / CLOB read/write)   │
└─────────────────────────────────────────────┘
```

### 推奨技術スタック
- **言語**: Python（py-clob-client対応、データ処理ライブラリ豊富）
- **DB**: SQLite（初期）→ PostgreSQL（スケール時）
- **メッセージング**: Redis（キューイング）
- **監視**: 簡易Webダッシュボード（Streamlit or Flask）+ Telegram通知
- **CI/CD**: なし（単一サーバー運用）
- **ホスティング**: VPS（US or JP、latency要件に応じて選択）
