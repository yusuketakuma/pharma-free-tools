## 1. ドメイン特化能力強化

### 1.1 基本戦略

**最優先順位:**
1. **dss-manager** - 既に専属名があり、反復オペレーション化しやすい
2. **pharma-free-tools** - 薬局/薬剤師文脈の専門知識が必要
3. **sidebiz/polymarket** - 市場監視・仮説評価のテンプレ整備

**domain-packの標準構成:**
- verification commands
- known failures  
- fix patterns
- decision axes
- procedure templates
- escalation boundaries

### 1.2 dss-manager 用 domain-pack (最優先)

**想定反復作業:**
- repo状態確認
- 不具合再現と影響範囲切り分け
- 修正前後の差分確認
- 既知障害との照合
- リリース前の最小検証
- 運用上の優先順位整理

** verification commands:**
```bash
# 開発サーバ起動
npm run dev

# 単体/統合テスト  
npm run test
npm run test:integration

# lint/typecheck/build
npm run lint
npm run type-check
npm run build

# 主要ユースケースのsmoke test
npm run test:e2e -- --spec smoke.spec.ts

# データ整合性確認コマンド
npm run db:check-consistency
```

**known failures:**
- よく壊れる画面/処理 (認証フロー、支払いフロー)
- 外部依存の不安定ポイント (決済API、メール送信)
- seed/migration/env mismatch
- stale cache/build artifact起因の不具合

### 1.3 pharma-free-tools 用 domain-pack

**想定反復作業:**
- 薬局業務フローに沿った機能要件整理
- 用語・データ項目・UI文言の妥当性確認
- 現場オペレーションに合うかのレビュー
- リスク高い誤解を招く表現の除去

**verification commands:**
- フォーム/入力系の回帰確認
- 主要画面の文言確認手順
- CSV/PDF/帳票系の生成確認
- role/permissionごとの動作確認

## 2. 収益化戦略 - 小さく検証可能なアプローチ

### 2.1 基本原則

**最重要: 最初は小さく検証可能な1件に絞る**
- 施策数を増やすより、短期間で当たり外れが判定できる形にすること
- 学習速度を上げることを優先する
- 「何をやるか」より「検証可能性」を優先

### 2.2 初回仮説例

**仮説:** 「薬剤師・小規模事業者向けの、日次業務を軽くする"記録/確認の省力化テンプレート"を、まずは1種類だけ提供すると売れる」

**具体化:**
- 1つの業務フローに絞ったテンプレート
- チェックリストと簡単な自動化手順
- 販売物は"総合支援"ではなく"1業務の時短キット"

### 2.3 選択基準

**この仮説を選んだ理由:**
- 既存の専門性に接続しやすい
- 顧客の課題が「毎日/毎週発生する面倒」に寄るため価値が伝わりやすい
- 実装コストを抑えやすく、最初の販売テストが速い
- 広げる誘惑を抑えられる

### 2.4 捨てたもの

- 複数業界への同時展開
- いきなり高額商材化
- フル自動化SaaS化
- 情報発信の大量生産を先行すること

## 3. エージェント運用の教訓

### 3.1 重要な問題パターン

#### A. status.md 未作成問題
**事象:** 39回連続でstatus.md未作成（プログラマーエージェント）
**対応:** 
- 優先度0強制指示 → テンプレート提供 → T219停止 → 成長評価格下げ
- 全タスクブロック発動 → ブロック解除条件にstatus.md存在を組み込み
- 結果: 9回目でようやく作成、技術的問題の可能性が高い

**教訓:** 
- 管理報告タスクは技術者にとって優先度が低くなりがち
- 「タスクの完了定義に組み込む」構造的強制も効果が出なかった
- 権限縮小（全タスクブロック）という組織的措置が最終手段

#### B. 波及チェック漏れ
**事象:** ポータルに10件追加したが、ヘッダー数値を同時更新しなかった
**対応:** 波及チェックの習慣化を指示 + 新規ツール追加チェックリスト作成
**教訓:** 変更の波及先を事前にリスト化する工程を標準化すべき

#### C. 分析→実行の不足
**事象:** CSV検証で問題を指摘したが、具体的対応案まで踏み込まなかった
**対応:** 「検証結果 + 実務アクション案」のセット出力を義務化
**教訓:** 分析で終わらず、実務への橋渡しまでが役割

#### D. 着手速度の格差
**事象:** homecareが12分で3タスク完了、sidebizが47分で0タスク着手
**対応:** 分析→推奨→実装の全ループを一サイクルで回すよう指示
**教訓:** 複数サイクルの明示的指摘が改善に有効、着手速度の差を数値でフィードバック

### 3.2 成功パターン

#### A. 「詳細版→運用版→印刷版」の3段階凝縮
**適用例:** テリパラチド対応
- アクションキット（詳細版）→ 3日間アクションプラン（運用版）→ 印刷用チェックリスト（印刷版）
- 情報量を段階的に絞り込みながら実務適用性を高める
- 他の緊急案件でも横展開可能

#### B. 定量的分析の自律化
**事例:** sidebizが実ファイルgrepによる品質検証を自律的に実施
- 「報告値の信頼性」問題は検証プロセスの自動化で構造的に解決可能
- 一度指摘→検証方法の提示→自律導入という教育フローが有効

#### C. 深夜帯dispatch設計の最適化
**静音モード（23:00-08:00）向け:**
- タスク粒度: 翌朝準備に特化した軽量タスク（整合性チェック・テンプレート準備・TODO整理）
- 目的: 翌朝のアクション開始をスムーズにする「前段取り」
- 避けるべき: 新規分析・大量ファイル処理など深夜に不要な重量タスク

### 3.3 品質管理

#### A. 完了証跡の問題
**原則:** `metric-claim-verification-checklist.md` に基づき、主証跡 + スポットチェック + 証跡メモをセットにしない限り、完了扱いにしない

**状態分離:**
- `review-approved` / `apply-applied` / `apply-blocked` / `manual_required` / `pending_artifact` / `effect-confirmed` を混同しない
- `done` や `complete` を書く前に、意図した効果がファイル・ログ・サンプルで確認できるかを必ず見る

#### B. 数値目標の明示
**効果:** 
- sidebizのcrosslink Batch1で「アウトバウンドリンク0のファイルを37→25以下」と設定し22を達成
- 目標数値をdispatchに明記することで成果の達成/未達成が一目瞭然

## 4. 自律開発のフレームワーク

### 4.1 OpenClaw vs Claude Code の役割分担

#### OpenClaw (control plane)
- タスク計画・仕様整理
- Claude Codeへの指示生成・ディスパッチ
- 実行結果の要約・報告
- 取締役会への議題作成
- 進捗管理・ボトルネック特定

#### Claude Code (execution plane)
- 複数ファイルの変更が必要な実装タスク
- テストの作成・実行・結果確認
- repo全体にまたがる調査・分析
- バグ修正・リファクタリング・コードレビュー
- 新機能の実装・パフォーマンス改善

### 4.2 Claude Code呼び出しトリガー条件

**自律的にClaude Codeを実行すべき場合:**
1. 複数ファイルの変更が必要な実装タスク
2. テストの作成・実行・結果確認
3. repo全体にまたがる調査・分析
4. バグ修正・リファクタリング・コードレビュー
5. 新機能の実装・パフォーマンス改善

### 4.3 Heartbeatの改善

**改善前:** 抽象的すぎるprompt
**改善後:** 具体的な選択肢を持つheartbeat
- noop: 対象なし・異常なし
- task_dispatch: Claude Code経由で実行すべき改善タスクを特定・ディスパッチ
- progress_report: 直近のClaude Code実行結果を確認・まとめ
- signal_only: Claude Codeのauth状態・lane healthに問題があればsignal

## 5. プロダクト運営のフレームワーク

### 5.1 役割分担モデル

#### 受理確認担当
- 依頼の受理有無を確認する
- 受理済みなら「受理証跡」を残す
- 未受理なら差し戻し理由を記録する
- 入口の状態を固定し、後工程に未確定情報を流さない

#### バックログ整理担当
- 受理済み依頼を棚卸しする
- 重複、粒度不足、期限切れ、依存関係不明を洗い出す
- stale backlogを検出して、保留/再確認/廃棄/再分割に振り分ける
- 実行待ちの状態を、見える化された一覧へ変換する

### 5.2 状態分離の原則

この運用では、状態を混ぜない。

- **入口状態:** 依頼が届いた / 受理した / 差し戻した
- **整理状態:** 分解済み / 保留 / 期限切れ / 重複 / 再確認待ち  
- **実行状態:** 着手可 / 進行中 / 完了 / 失敗
- **記録状態:** proof-pathに保存済み / 未保存

**ルール:**
- 入口の曖昧さを、実行状態に持ち込まない
- 整理担当は「着手可」を作るのであって、実行そのものは担わない
- 受理確認担当の判断と、バックログ整理担当の判断を同一メモに混ぜない

### 5.3 Proof-path の確保

各判断は、後から追える形で残す。

**最低限の証跡:**
- 元依頼の本文または要約
- 受理/差し戻しの判断理由
- 整理時の分類理由
- 参照したファイルパスやメモの場所
- 最終的な振り分け先

**推奨保存先:**
- `reports/`
- `reports/runbooks/`
- `reports/inbox/`
- `reports/backlog/`

### 5.4 Stale backlog 対応基準

以下は stale backlog として優先的に処理する。

**Stale backlog基準:**
- 期限が切れている
- 何度も再読しても着手条件が変わらない
- 依存先が未確定で停滞している
- 情報不足のまま放置されている
- 同種の新規依頼に埋もれて再発見されにくい

**対応方針:**
1. まず受理証跡を確認する
2. 次に整理可能かを判定する
3. 整理不能なら再確認依頼へ戻す
4. 実行価値が薄いものは廃棄候補にする
5. 重要だが未着手のものは、期限付きで再掲する

### 6.1 即時実施すべきこと

1. **dss-manager用のdomain-pack作成**
   - 最低限のverification commands定義
   - 既知の障害パターン整理

   - 業務フローに沿った確認手順
   - 用語辞書とチェックリスト

3. **深夜帯dispatch設計の標準化**
   - 翌朝準備向け軽量タスクのテンプレ
   - スリープ移行後の巡回間隔延長（30分→60分）

### 6.2 短期での改善点

1. **完了証跡の徹底**
   - 各タスク完了時に具体的な証跡を要求する
   - 数値目標と達成結果のペアリングを標準化

2. **着手速度の改善**
   - homecareの成功パターンを他エージェントに横展開
   - 「分析→実装」の全ループを一サイクルで回る習慣定着

3. **品質検証の自律化**
   - 報告値の信頼性問題を構造的に解決
   - 自律的な品質検証プロセスを各エージェントに導入

### 6.3 中期的な目標

1. **ドメイン特化エージェントの網羅化**
   - dss-manager → pharma-free-tools → sidebiz/polymarket
   - 各domain-packの成熟度を評価・改善

2. **収益化仮説の検証**
   - 初回仮説の市場ニーズ評価を実施
   - 検証結果に基づく次の仮説調整

3. **運営フレームワークの標準化**
   - プロダクト運営の役割分担を複数プロジェクトに適用
   - 状態管理とproof-pathの仕組みを定着
