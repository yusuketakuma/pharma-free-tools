# プロダクト運営チェックリスト - 受理確認/バックログ整理

最終更新: 2026-03-29
対象: プロダクト運営本部の受理確認担当 + バックログ整理担当

---

## 📋 概要

本ドキュメントは、プロダクト運営本部の受理確認担当とバックログ整理担当が使用するチェックリストとフレームワークです。依頼を迷子にせず、実行可能な状態へ整え、滞留を早期に見える化する運用を標準化します。

### 役割分担の基本原則

- **受理確認担当**: 入口の状態を固定し、後工程に未確定情報を流さない
- **バックログ整理担当**: 実行待ちの状態を、見える化された一覧へ変換する
- **重要なルール**: 状態を混ぜない、判断ログは分離して保存

### 1.3 受理判断基準詳細

#### 受理条件（満たす必要あり）
| 項目 | 評価基準 | 評価方法 |
|------|----------|----------|
| 目的明確性 | 目的が具体的に記載されているか | 内容のレビュー |
| 範囲適切性 | 必要な範囲が過不足なく定義されているか | 要求内容の分析 |
| 情報充足性 | 判断に必要な情報が揃っているか | 必要項目の確認 |
| 優先度設定 | 優先度が合理的に設定されているか | 業務影響度の評価 |
| 担当者明確性 | 担当者が明確になっているか | 組織構造との照合 |

#### 差し戻し条件（いずれかを満たす）
| 差し戻し理由 | 具体的な状況 | 対応方法 |
|-------------|-------------|----------|
| 情報不足 | 必要な情報が欠けている | 情報追加を要求 |
| 範囲不明確 | 要求範囲が曖昧 | 範囲の明確化を要求 |
| 優先度不当 | 優先度が現実的でない | 優先度の再設定 |
| 担当者不明 | 処理担当者が特定できない | 担当者の選定を要求 |
| 重複依頼 | 重複した依頼がある | 重複の解消を要求 |


- `#[種別]` 例: `#新機能`, `#修正`, `#質問`

### 2.3 Stale Backlog 対応基準

#### Stale Backlog 診断チェックリスト

| 診断項目 | 診断基準 | 診断結果 | 対応方針 |
|----------|----------|----------|----------|
| 期限切れ | 設定期限を超過している | [はい/いいえ] | 期限延長 or 廃棄 |
| 再読不能 | 何度読んでも着手条件が変わらない | [はい/いいえ] | 再確認 or 廃棄 |
| 依存停滞 | 依存先が未確定で長期停滞 | [はい/いいえ] | 依存先の確認 or 分解 |
| 情報不足 | 必要情報が不足して放置 | [はい/いいえ] | 情報収集 or 廃棄 |
| 競合埋没 | 同種の新規依頼に埋もれて再発見困難 | [はい/いいえ] | 統合 or 廃棄 |

#### Stale Backlog 対応プロセス

##### ステップ1: 受理証跡の確認
- [ ] 元依頼の確認
- [ ] これまでの対応履歴の確認
- [ ] 関連依頼の確認

##### ステップ2: 整理可能かの判定
- [ ] 情報があれば整理可能か判断
- [ ] 依存関係が解消されるか確認
- [ ] 実行価値があるか再評価

##### ステップ3: 状態遷移の実施
**整理可能の場合:**
- [ ] 必要情報の追加を依頼
- [ ] 依存先の確認を実施
- [ ] 優先度の再設定
- [ ] 状態を「実行可能」に変更

**整理不能の場合:**
- [ ] 再確認依頼として分離
- [ ] 期限付きで再掲
- [ ] 廃棄候補としてマーク
- [ ] 担当者への通知

##### ステップ4: 証跡の保存
- [ ] 判断理由の記録
- [ ] 状態変化の記録
- [ ] 次アクションの設定
- [ ] 関連ファイルの整理

## 4. トラブルシューティング

### 4.1 受理プロセスのトラブル

#### 問題1: 受理基準の曖昧さ
**現象**: 担当者間で受理判断にばらつきがある
**原因**: 受理基準の定義が不十分
**対策**:
- [ ] 受理基準の明文化
- [ ] 判断事例の共有
- [ ] 担当者間の合意形成
- [ ] 定期的な基準レビュー

#### 問題2: 受理証跡の不備
**現象**: 判断理由や担当者の記載が不十分
**原因**: チェックリストの不備
**対策**:
- [ ] 証跡テンプレートの改善
- [ ] 必須項目の追加
- [ ] 自動生成の導入
- [ ] 定期的なレビュー

#### 問題3: 差し戻しの遅延
**現象**: 差し戻し依頼の修正が遅れる
**原因**: 修正指示が不明確
**対策**:
- [ ] 差し戻し理由の具体化
- [ ] 修正指示のテンプレ化
- [ ] 期限の明確化
- [ ] フォローアップの強化

### 4.2 整理プロセスのトラブル

#### 問題1: Stale Backlogの放置
**現象**: 期限切れや情報不足の依頼が放置される
**原因**: 定期的なチェックの不備
**対策**:
- [ ] 自動的な stale 検出の導入
- [ ] 定期的なレビューの強化
- [ ] アラートシステムの導入
- [ ] 優先順位の自動再評価

#### 問題2: 状態遷移の誤り
**現象**: 状態が正しく遷移しない
**原因**: 遷移ルールの不明確
**対策**:
- [ ] 状態遷移ルールの明文化
- [ ] 遷移条件の厳格化
- [ ] 自動遷移の導入
- [ ] 定期的な監査

#### 問題3: 証跡の不整合
**現象**: 証跡情報が実態と異なる
**原因**: 手動入力のエラー
**対策**:
- [ ] 自動証跡生成の導入
- [ ] 一貫性チェックの強化
- [ ] 定期的なデータ検証
- [ ] ファイルシステムの監視

### 4.3 チーム運営のトラブル

#### 問題1: 担当者の負荷不均衡
**現象**: 特定の担当者に負荷が集中する
**原因**: 担当者割り当ての不適切

- **測定方法**: 定期的な自動検出

- **測定方法**: 整理結果の分析
- **更新頻度**: 毎週
