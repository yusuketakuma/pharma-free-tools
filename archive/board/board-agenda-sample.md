# 取締役会本会議議題例 - 論点配置判定

## 選択論点（最大3件）

### 1. Q2収益計画の再調整
- **背景**: 市場環境変化を受けた収益見直し。競合動向と需要予測の変更を反映
- **配置**: OpenClaw 完結
- **理由**: plan_onlyの調整作業であり、read_only調査と軽量調整で完了可能。数値調整と計算ロジックの変更のみ

### 2. 新規API機能のリファクタリング提案
- **背景**: 既存APIの複雑性改善とパフォーマンス向上のため、大規模なリファクタリングが必要
- **配置**: Claude Code execution
- **理由**: 複数ファイル変更・コード修正・テスト実行が必要なため。backend-architectの専門知識が必要

### 3. セキュリティポリシー更新
- **背景**: 新たなリスク対応とコンプライアンス要件に基づくポリシー改定
- **配置**: OpenClaw 完結
- **理由**: read_only調査と文章整備のみで完了可能。法的要件の確認とポリシードキュメントの更新

## 論点配置判定詳細

   - **理由2**: リファクタリングに伴うパフォーマンステストが必要

2. **セキュリティポリシー更新**
   - **理由1**: read_only調査が主作業
   - **理由2**: 法的要件の確認とドキュメント更新のみ

## 実行面の配置判断理由

### 判断プロセス
1. **論点タイプ分類**: 各論点をread_only, plan_only, code_changeに分類
2. **重量評価**: 複雑さ・影響範囲・リスクを3段階評価
3. **専門性判定**: 必要な専門知識の有無を確認
4. **配置決定**: 前述の条件に基づき配置を決定

### 重要考慮事項
1. **リファクタリングの特殊性**: 
   - 単純な修正ではなく、大規模な変更とテストが必要
   - パフォーマンス改善を伴うため専門家の関与が必須

2. **計画調整の軽量性**:
   - 数値修正と計算ロジックの変更のみ
   - 重い調査や複雑な変更は不要

3. **ポリシー更新の特性**:
   - コンプライアンス要件の確認が主
   - ドキュメントの更新作業が中心
   - 技術的複雑性は低い

### 判断根拠の詳細

#### Claude Code executionの判断理由
- **必要性**: リファクタリングは複数ファイルに影響
- **専門性**: API設計とパフォーマンス最適化の専門知識が必要
- **作業内容**: コード変更・テスト・パフォーマンステストが必要
- **リスク**: 不適切なリファクタリングはシステム全体に影響

#### OpenClaw完結の判断理由
- **作業量**: 計画調整とポリシー更新は軽量作業
- **専門性**: 特定の専門知識は不要
- **複雑さ**: 単純な調整と文章作業のみ
- **リスク**: 軽微な調整のため、control planeで十分管理可能

### 配置変更のケーススタディ

#### Case 1: 配置変更が必要なケース
```
論点: 小規模なバグ修正
- 初期配置: OpenClaw完結（軽微な修正と判断）
- 再判定: Claude Code execution（複雑な依存関係があるため）
- 理由: 修正範囲が想定より広く、複数ファイルに影響するため
```

#### Case 2: 軽量化可能なケース
```
論点: 大規模調査プロジェクト
- 初期配置: Claude Code execution（調査の専門性が必要と判断）
- 再判定: OpenClaw完結（調査範囲がread_onlyに限定可能と判明）
- 理由: 担当者調査とドキュメントレビューのみで完了可能なため
```

### 配置のベストプラクティス

1. **初期配置は conservative に**
   - 複雑性が不明な場合は一度Claude Codeに振り分け
   - 結果に基づいて再判定

2. **配置変更は明示的に**
   - 変更の理由と根拠を記録
   - 影響範囲を評価して変更

3. **振り分け基準の継続改善**
   - 過去の実績から基準を見直し
   - 新しいパターンに対応して基準を更新
