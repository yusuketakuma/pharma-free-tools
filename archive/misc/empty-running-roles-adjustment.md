# 空回り担当の調整計画

**調整日時**: 2026-03-28 14:00 GMT+9  
**対象担当**: 2つの空回りエージェント

## 対象担当の状況

### 🔴 receipt-delivery-reconciler (受理確認担当)
- **状態**: 13.5時間待機状態
- **問題**: 長時間アイドル状態
- **影響**: リソースの無駄遣い
- **優先度**: 高

### 🟡 backlog-triage-clerk (バックログ整理担当)
- **状態**: diminishing_returns状態
- **問題**: 収益逓減状態
- **影響**: 効率低下
- **優先度**: 中

## 調整アクション

### Priority 1: receipt-delivery-reconcilerの停止
**アクション**: cron停止 or 間隔延長
- **現在**: 13.5時間待機 = 非常に長い間隔
- **推奨**: 停止 or 24時間延長（1日1回程度）

**理由**:
- 13.5時間の待機は不合理
- 受理確認はリアルタイム性を必要としないタスク
- リソース解放による全体効率化

### Priority 2: backlog-triage-clerkの最適化
**アクション**: 効率改善 or 間隔調整
- **現在**: diminishing_returns状態
- **推奨**: 効率分析 or 週1回程度の頻度に変更

**理由**:
- バックログ整理は逓減効果が発生
- 頻度を下げて効率を改善
- 効率分析のための追加指標導入

## 具体的な調整計画

### receipt-delivery-reconciler
**調整前**:
- 実行間隔: 13.5時間
- 状態: 待機中
- リソース消費: 高

**調整案**:
```
Option A: 完全停止
- 状態: 停止
- トリガー: 手動実行 or 他エージェントからの呼び出し

Option B: 間隔延長  
- 実行間隔: 24時間
- 状態: 定期的に実行
- リソース消費: 低

Option C: 条件付き実行
- 実行条件: 受理数が一定閾値を超えた場合のみ
- 状態: イベント駆動
- リソース消費: 最小
```

**推奨**: Option C (条件付き実行)

### backlog-triage-clerk
**調整前**:
- 実行間隔: 高頻度（推定1時間）
- 状態: diminishing_returns
- 効率: 低下傾向

**調整案**:
```
Option A: 頻度変更
- 実行間隔: 週1回 or 月1回
- 状態: 定期的に実行
- 効率改善: 期待

Option B: 効率分析
- 分析項目: どの程度の頻度が最適か
- 状態: 分析中
- 効率改善: データに基づいた最適化

Option C: 職務再定義
- 新たな職務: バックログの品質向上
- 状態: 職務変更
- 効率改善: 質の向上
```

**推奨**: Option B → Option A (分析後に頻度変更)
