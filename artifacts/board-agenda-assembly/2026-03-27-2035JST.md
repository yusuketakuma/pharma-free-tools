## 1) 実行系の配車・滞留・完了率
- 論点: OpenClaw の control plane で、現在の queue / rebalance / approval の滞留がないか。必要なら誰をどこへ振るか。
- 判定: **OpenClaw 完結**
- 理由: 調整・優先順位・配車判断は control plane の責務で、コード変更や検証を伴わない。

## 2) 主要課題の処置方針（最大3件）
- 論点: 直近の詰まりを 3 件までに圧縮し、各件について「何を決めるか」「何を実行するか」「誰が持つか」を明確化する。
- 判定: **案件ごとに分岐**
  - read_only / plan_only / short report / lightweight coordination → **OpenClaw 完結**
  - repo 調査 / 複数ファイル変更 / テスト / 実装 / refactor → **Claude Code execution plane**
- 理由: 仕様確認や整理は OpenClaw で十分だが、実装・検証は Claude Code に寄せた方が速く安全。

## 3) ルール・記録・再発防止
- 論点: 今回の判断を runbook / proposal / memory / changelog のどれかに残し、次回以降の再利用可能な形へ固定する。
- 判定: **OpenClaw 完結**
- 理由: 記録・標準化・運用定着は control plane の役割で、実装作業は不要。

## Claude Code execution へ回す論点
- repo-wide 調査が必要な案件
- 複数ファイル変更が必要な案件
- テスト実行が必要な案件
- 実装 / refactor / heavy verification が必要な案件

## OpenClaw 完結でよい論点
- agenda seed の圧縮
- 進行役の裁定
- 配車・優先順位・軽い確認
- short report / coordination
- 記録先の決定

## 実行面の配置判断理由
- OpenClaw は control plane なので、論点の整理・裁定・配車・記録を担う。
- Claude Code は execution plane なので、repo 調査、コード変更、テスト、実装、refactor を担う。
- 重要なのは「決める場所」と「実際にやる場所」を分離し、危険な変更や重い実行を OpenClaw から直接やらないこと。
