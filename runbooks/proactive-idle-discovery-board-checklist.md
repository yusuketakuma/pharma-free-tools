# Proactive Idle Discovery Board Checklist

目的: 自律探索タスクを `CEO ↔ 取締役会 ↔ 実行エージェント` 構造で安全に回し、候補の乱発・重複・未整理報告を防ぐ。

## 0. 前提
- ユーザー報告は原則 board review 済みの内容のみ
- 実行エージェントの生判断をそのまま報告しない
- 1回の探索で提示候補は最大3件
- 実際の自動着手は最大1件
- 高リスク案件は自動適用しない

## 1. まず確認すること
### 1-1. 重複抑止
- [ ] 直近24時間の memory / report / proposal を確認した
- [ ] 同種テーマの再着手なら差分を明記した
- [ ] 新規性が薄い場合は `更新なし / 見送り / 既存成果物へ統合` のいずれかに倒した

### 1-2. 探索優先順位
次の順で候補を探す。
1. 既存タスクの停滞・詰まり解消
2. 既存調査の成果物化
3. 収益化に近い改善
4. 反復作業の仕組み化
5. repo / domain 専用の改善候補
6. 新規機会の一次探索

### 1-3. 探索禁止領域
- [ ] Telegram 設定変更ではない
- [ ] auth / trust boundary / routing / approval の根幹変更ではない
- [ ] 課金や対外送信を伴わない
- [ ] 既存 high-priority task を壊す横入りではない
- [ ] 価値が曖昧な一般論調査ではない

## 2. Board 役割ごとの見るポイント
### Board Visionary
- 構造的に効くか
- 将来の自動化・再利用に繋がるか
- 小さな一手で大きなレバレッジがあるか

### Board User Advocate
- 導入負荷は低いか
- 使う人にとって分かりやすいか
- 運用時の認知負荷を減らすか

### Board Operator
- 今日できる最小実行可能案は何か
- どこまでを自動、どこからを manual review にするか
- 依存関係なしに前進できるか

### Board Auditor
- 可逆性はあるか
- 失敗時の影響範囲は限定できるか
- protected path / 本番波及 / 破壊変更に触れないか

### Board Chair
- 論点を整理したか
- 採用 / 保留 / 却下 / 追加調査 を明確にしたか
- ユーザー報告が board review 後の内容だけになっているか

## 3. 高リスク判定
次のいずれかなら deep review 必須。
- auth / trust boundary / routing / approval / Telegram 根幹変更
- 破壊的変更
- 複数 repo / 複数 agent / 本番運用へ広く波及
- 取り返しがつきにくい変更

高リスク案件の結論は次のいずれかに止める。
- 採用
- 保留
- 却下
- 追加調査

## 4. 自動着手の条件
以下をすべて満たす場合のみ自動着手してよい。
- [ ] 低リスクな内部作業
- [ ] 可逆的
- [ ] protected path を触らない
- [ ] 外部送信なし
- [ ] 他の high-priority task を妨げない
- [ ] 成果物を残せる

## 5. 成果物ルール
自律探索で見つけた仕事は、可能なら次のいずれかを残す。
- report
- runbook
- checklist
- proposal
- template
- memory 候補

成果物を残せない場合は、理由を報告に明記する。

## 6. 標準レポート形式
```md
## 結論
- 何を採用 / 見送りしたか

## 今回見つけた候補（最大3件）
1. 候補名
2. 候補名
3. 候補名

## board の採否判断
- 候補1: 採用 / 保留 / 却下 / 追加調査
- 候補2: ...
- 候補3: ...

## 実際に着手したもの（最大1件）
- 着手内容
- 低リスク判定理由

## 残した成果物 / 差分
- ファイルパス
- 要点

## 見送った理由
- 候補ごとの見送り理由

## 次アクション
- 次回の探索起点
- manual review が必要な点
```

## 7. 推奨運用
- 通常通知は定時報告へ集約
- 同じテーマを繰り返すより、既存成果物へ統合する
- 調査だけで終わらせず、可能ならテンプレート化・チェックリスト化・提案書化する
- 実行エージェントは探索と素材出しに使い、最終判断は必ず board で行う
