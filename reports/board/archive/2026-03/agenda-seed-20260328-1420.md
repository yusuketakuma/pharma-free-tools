**理由**: 現在、OpenClawは夜間もcron・サブエージェント・自動改善提案などを自律実行しているが、その実行結果の確認は個別ログやセッションログの検索に依存している。異常が起きていても発見が遅れ、手戻りや認知負荷が増える。SOUL.mdに定める「ゆうすけが寝ていても回る仕組み」の前提として、回っていることが確信できる状態が不可欠である。可視化がなければ自律運転はブラックボックスであり、信頼も品質も担保できない。

**リスク**: ダッシュボード構築自体が新規開発工数を吸収し、他タスクを圧迫する可能性。過剰なアラート設定によるアラート疲れ（ノイズ過多で実質的に無視される状態）。監視対象メトリクスの定義が不十分だと、見えているのに判断できないダッシュボードになる。認証・アクセス制御の設計不備による情報漏洩リスク（protected pathに該当するためmanual review必須）。プライバシー・セキュリティ面でログに機密情報が含まれる場合の取り扱い。

**理由**: 現在、エージェント群は継続的に改善・自動化を進めているが、その効果がゆうすけの時間削減としてどの程度貢献しているかを定量的に把握する仕組みがない。効果が見えなければ、優先順位の判断や不要な自動化の棚卸しができず、結果的にゆうすけのリソース配分が最適化されない。AGENTS.md/SOUL.mdが掲げる「ゆうすけの成果を最大化し、負担を減らす」という目的に対し、定期的な効果測定は不可欠。

**リスク**: ROIの算定基準（想定手作業時間の見積もり精度）が曖昧になり、形式だけのレビューになる。レビュー自体がゆうすけの負担になる（→レビューはエージェント側で集計し、ゆうすけは確認のみ）。自動化タスクの分類・追跡のための追加ログ設計が必要。

**次アクション**: 自動化タスクのROI算出テンプレート（想定手作業時間/自動化時間/頻度）を作成する。2026年Q1分の自動化タスクについてバックフィルで試算する。レビュー結果をメモリ・レポートとして蓄積する形式を定義する。

**結論**: エージェントが自律実行したタスクの成果物に対し、定期レビューを行う担当者（または自動チェックパイプライン）を定義し、品質ゲートを設けることを承認する。

**理由**: OpenClawの自律運転タスク（定期提案、自動整理、メモリ更新など）は増加傾向にあるが、成果物の品質チェックが個別エージェントの判断に委ねられている。誤った提案が残存したり、低品質な出力が蓄積されると、後続の意思決定やエージェントの動作品質に悪影響を及ぼす。継続的な信頼性を担保するには、自動出力に対するレビュー層が不可欠。

**次アクション**: 自律運転タスクの成果物分類（要レビュー / 自動チェック済み / 免除）の基準を策定する。低リスク出力に対する自動チェックパイプラインの設計を先行させる。高リスク出力（提案・設定変更・外部送信）のレビュー担当とSLAを定義する。

**理由**: OpenClaw + Claude Codeの連携インフラは整備済みだが、実プロジェクトへの適用は未承認の状態にある。コード調査・影響分析・テスト結果読解などread-onlyタスクは低リスクで自動化効果が高い。ゆうすけの判断負荷を減らし、開発サイクルを加速させるための最小ステップとして妥当。パイロットで稼働実績を蓄積すれば、writeタスクの段階的自動化判断の材料になる。

**リスク**: Claude Code実行時のsubscription認証の安定性（auth drift時のfail-closed挙動でタスクが滞留）。read-onlyタスクであってもlarge repo全体の調査は実行時間・トークン消費が大きくコスト予測困難。パイロット対象プロジェクトの選定を誤ると稼働実績としての評価が歪む。

**次アクション**: パイロット対象プロジェクトをゆうすけに最終選定してもらう。対象プロジェクト内のread-onlyタスク種別をカタログ化。Claude Code lane（acp_compat）のhealth監視を有効化し、auth drift検知時のアラートをTelegramに通知するよう設定。2週間のパイロット運用を実施し、実行成功率・平均所要時間・コストをメトリクスとして記録。パイロット結果を基に、writeタスク自動化の可否判定を行う次回取締役会を設定する。
