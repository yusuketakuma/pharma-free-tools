# Supervisor Core 判断品質レビュー — 2026-03-25

## 結論
Supervisor Core の判断は **安全で具体的** だが、**重複がやや多く、実務インパクトの押し出しが弱い**。
今回の直近出力は、観測精度は上がった一方で、提案の中心が引き続き queue telemetry / stale-report 系に寄っている。

## 判断品質レビュー
- **具体性**: 4/5
  - `waiting_auth=476` / `waiting_manual_review=343`、最古/最新 mtime、top prefixes、invalid JSON、24h delta まで出しており、根拠は十分。
- **実行可能性**: 5/5
  - read-only スナップショットで、変更リスクが低い。すぐ実施できる形になっている。
- **重複の少なさ**: 2/5
  - `queue telemetry snapshot` と `stale-report detection` が、前回提案とかなり近い。新規性が薄い。
- **優先順位の妥当性**: 3/5
  - 監視強化は妥当だが、既に telemetry は揃い始めているので、次は「観測」より「詰まりの解消」へ寄せた方がよい。
- **実務インパクト**: 3/5
  - 現状把握には効くが、滞留解消や意思決定の速度改善への直結度はまだ中程度。

## 弱い点
1. **観測提案の反復**
   - 似た内容の queue 可視化が続いていて、判断の差分が小さい。
2. **Impact-first ではない**
   - 何が最も backlog を減らすか、どの提案が最短で詰まりを解くかの順位付けが弱い。
3. **次アクションが抽象寄り**
   - telemetry は良いが、その後の triage / remediation への橋渡しが薄い。

## 原因仮説
- **安全寄り最適化が強い**: 低リスク提案に寄るため、観測改善が繰り返されやすい。
- **提案間の重複検出が弱い**: 前回の提案を十分に踏まえた差分判定がない。
- **評価軸に実務インパクトの重みが足りない**: 実行しやすさを優先しすぎて、効果の大きさが後回しになっている。

## 改善案
1. **重複チェックを必須化**
   - 直近1〜2回の提案と同系統なら、「新しい根拠」か「新しい対象範囲」がない限り再提案しない。
2. **Impact-first の順位付けを追加**
   - `実行容易性` だけでなく、`backlog削減効果` / `blocked件数への効き` / `再発防止効果` を必須比較にする。
3. **1提案 = 1次アクション + 1成功条件**
   - 抽象提案を減らし、次に何をやるかと、成功したかどうかを1行で判定できる形に固定する。

## 前回との差分
- 前回は「queue telemetry を取るべき」という**提案レベル**が中心だった。
- 今回は実測スナップショットがあり、**観測の具体性は上がった**。
- ただし、結論の方向はほぼ同じで、**重複感は残る**。
- 24h delta が 0 で、バックログが新規増加よりも**停滞・滞留型**だと確認できた点は前進。

- 直近提案との重複判定を入れて、同系統の再提案を減らす。
