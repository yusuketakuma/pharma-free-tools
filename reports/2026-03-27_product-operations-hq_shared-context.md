# 本部 shared context

- 日付: 2026-03-27
- 本部: プロダクト運営本部
- 対象運用: 受理確認担当 + バックログ整理担当を使った運営ラインの初回運用
- 目的: 受理された依頼を迷子にせず、実行可能な状態へ整え、滞留を早期に見える化する

## 役割分担

### 受理確認担当
- 依頼の受理有無を確認する
- 受理済みなら「受理証跡」を残す
- 未受理なら差し戻し理由を記録する
- 入口の状態を固定し、後工程に未確定情報を流さない

### バックログ整理担当
- 受理済み依頼を棚卸しする
- 重複、粒度不足、期限切れ、依存関係不明を洗い出す
- stale backlog を検出して、保留 / 再確認 / 廃棄 / 再分割 に振り分ける
- 実行待ちの状態を、見える化された一覧へ変換する

## state separation

この運用では、状態を混ぜない。

- 入口状態: 依頼が届いた / 受理した / 差し戻した
- 整理状態: 分解済み / 保留 / 期限切れ / 重複 / 再確認待ち
- 実行状態: 着手可 / 進行中 / 完了 / 失敗
- 記録状態: proof-path に保存済み / 未保存

ルール:
- 入口の曖昧さを、実行状態に持ち込まない
- 整理担当は「着手可」を作るのであって、実行そのものは担わない
- 受理確認担当の判断と、バックログ整理担当の判断を同一メモに混ぜない

## proof-path

各判断は、後から追える形で残す。

最低限の証跡:
- 元依頼の本文または要約
- 受理/差し戻しの判断理由
- 整理時の分類理由
- 参照したファイルパスやメモの場所
- 最終的な振り分け先

推奨保存先:
- `reports/`
- `reports/runbooks/`
- `reports/inbox/`
- `reports/backlog/`

## stale backlog 対応基準

以下は stale backlog として優先的に処理する。
- 期限が切れている
- 何度も再読しても着手条件が変わらない
- 依存先が未確定で停滞している
- 情報不足のまま放置されている
- 同種の新規依頼に埋もれて再発見されにくい

対応方針:
1. まず受理証跡を確認する
2. 次に整理可能かを判定する
3. 整理不能なら再確認依頼へ戻す
4. 実行価値が薄いものは廃棄候補にする
5. 重要だが未着手のものは、期限付きで再掲する

## 初回運用のゴール

- 入口と整理の責務分界を固定する
- proof-path を必須化する
- stale backlog を放置せず、必ず状態遷移させる
- 以後の運営ラインを再利用可能な形にする

## 今回の判断

- 受理確認は入口の純化を優先
- バックログ整理は滞留解消を優先
- 両者の判断ログは分離して保存
