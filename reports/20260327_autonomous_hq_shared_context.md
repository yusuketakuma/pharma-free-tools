# 本部 Shared Context｜仕様→実装→検証ライン 初回運用設計

- 日付: 2026-03-27
- 役割: 自律開発本部長
- 対象: 仕様→実装→検証ラインの初回運用設計
- 方針: まだ重い実装は行わず、**low-risk change を1本流す前提**で、運用の型を先に残す

## 1. 目的

初回運用で迷わないように、仕様・実装・検証の受け渡しを明文化する。
今回の成果は、機能追加そのものではなく、**安全に小さく流せる運用ラインの設計** である。

## 2. 重要な前提

- これは **apply の成立** を目指す作業であり、**success の保証ではない**
- 低リスクな変更だけを扱う
- protected path、認証、trust boundary、Telegram の本体設定は触らない
- 実装は最小限に留め、まずはテンプレート化・手順化を優先する

## 3. 初回運用の基本形

### 仕様
- 目的、範囲、非対象を1枚で書く
- 変更対象を「1本」に絞る
- 失敗時の戻し方を先に書く

### 実装
- 変更は小さく分割する
- 影響範囲が読めるものだけを実施する
- 仕様との差分を明示する

### 検証
- 成功条件と失敗条件を事前定義する
- 自動確認と人手確認を分ける
- 成果物を残してから次へ進む

## 4. 今回の low-risk change の扱い

今回は本番機能の追加ではなく、以下のような軽い変更を対象にする。

- 運用メモ
- 手順テンプレート
- 検証チェックリスト
- 受け渡し用の markdown

この層でラインを通し、次回以降に実装変更を安全に載せる。

## 5. 運用ルール

1. 仕様は先に固定する
2. 実装は最小単位にする
3. 検証条件を先に書く
4. apply と success を分けて記録する
5. 失敗しても学びが残る形にする

## 6. 初回の成果物定義

- 本部 shared context
- 本部長→取締役会報告
- 変更検証担当または DDS 担当への handoff

これらを毎回残すことで、運用が属人化しないようにする。

## 7. 主要リスク

- 仕様が曖昧なまま実装に入る
- apply できたことで成功扱いしてしまう
- 検証条件が後出しになる
- low-risk のつもりで高リスク変更に踏み込む

## 8. 推奨

今回の初回は、**ドキュメントと手順の型を先に固定し、その後に小さな変更を1本だけ流す** のがよい。
これにより、次回からの自律実行に再利用性が出る。
