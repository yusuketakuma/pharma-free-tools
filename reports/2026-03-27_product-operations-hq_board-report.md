# 本部長→取締役会報告

## 件名
受理確認担当とバックログ整理担当を使った運営ラインの初回運用を開始

## 結論
運営ラインの初回運用方針を定義した。今回は「受理確認」と「バックログ整理」を分離し、proof-path を残し、stale backlog を放置しない構造にする。

## 実施内容
- 本部 shared context を作成
- 受理確認担当とバックログ整理担当の責務を分離
- state separation の基準を明文化
- stale backlog の判定基準と処理方針を定義
- 各判断を追跡可能にする proof-path 方針を明記

## 観点別評価

### proof-path
良い点:
- 受理判断、整理判断、最終振り分けを分けて記録できる
- 後から「なぜその状態になったか」を追いやすい

リスク:
- 記録先が散ると証跡が追いにくくなる

対策:
- 保存先の命名規則を固定し、同じ構造で継続する

### state separation
良い点:
- 入口状態、整理状態、実行状態を混ぜずに扱える
- 依頼の曖昧さが実行工程へ漏れにくい

リスク:
- 分離が強すぎると担当間の受け渡しが増え、滞留しやすい

対策:
- 各状態の終了条件を短く明確にする

### stale backlog
良い点:
- 放置案件を先送りにせず、再確認 / 廃棄 / 再分割へ送れる

リスク:
- stale 判定が甘いと滞留が見えない

対策:
- 期限切れ、依存不明、情報不足を機械的に拾う

## 取締役会への要請
- この運営ラインを「初回の標準形」として承認したい
- 次回以降、同じ分離基準と proof-path 方針で運用を続けたい

## 今後の指示
- 受理確認担当: 受理/差し戻しの判断ログを固定化
- バックログ整理担当: stale backlog の棚卸しと再分類を最初の定常業務にする
