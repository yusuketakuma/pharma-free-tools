# guideline optimization for current system

## 実行したこと
- step3 の内容を、外部記事の例そのままではなく、現在の OpenClaw / Board / Claude Code / Telegram 運用に最適化する方向へ修正した。
- guideline 群に、現在の実運用で重要な論点を追加した。
- guidelines 自体を self-improvement 対象として扱う契約を明文化した。

## 最適化した要点
- Telegram は主操作面だが exec approval に向かない、という surface 制約を company-overview に反映した。
- いまの主課題を「状態レーン混同」「pending_artifact 長期化」「broad role 増殖」として明記した。
- CEO スタイルに、dense communication / sparse deliberation と「最後の詰まりを先に潰す」判断原則を追加した。
- ブランド guideline に、運用話でも人がすぐ判断できる表現を優先し、内容→改善→残件の順で報告する型を追加した。
- research protocol に、approval surface / routing / receipt / cadence のズレを先に疑う実運用ルールを追加した。
- top-posts-reference は SNS 投稿だけでなく、分かりやすい運用報告パターンも蓄積対象に広げた。

## self-improvement 契約の要点
- guideline / template / receipt / backlog / handoff / report の詰まりを改善対象にする。
- protected path / auth / approval / trust boundary / routing root / Telegram 設定は自動変更しない。
- 新しい問題は template → guideline → agent 定義 / 新役追加 の順で対処する。
- wording を変えるだけでなく、report / handoff / verification に波及したことを完了条件にする。

## 期待する改善
- 外部記事の generic ルールではなく、いまのたまAI運用のボトルネックに効く guideline 群になる。
- self-improvement が agent 増殖より先に guideline / template を育てる方向へ寄る。
