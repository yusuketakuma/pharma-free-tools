# DeadStockSolution ゼロベース再構築ロードマップ

作成日: 2026-02-26
対象: 既存機能を維持しつつ、運用しながら段階的に再構築する

## 0. 再構築の基本方針
- 業務停止なし（Strangler Fig パターンで段階移行）
- 既存API互換を維持し、クライアント影響を最小化
- 先に「安全性・監視性・テスト」を整備してから置換する

## 1. フェーズ別ロードマップ

### Phase 1: 土台整備（2026-03-01 〜 2026-03-21）
- ドメイン境界定義（Auth / Inventory / Exchange / Notification / Admin）
- API契約の固定（OpenAPI化、互換ポリシー定義）
- 観測基盤（構造化ログ、メトリクス、トレース、SLO）
- テスト戦略統一（unit/integration/e2e契約）

完了条件:
- 本番障害時の追跡がログ/メトリクスだけで可能
- 主要APIの契約テストがCIで常時実行

### Phase 2: データ・通知基盤再設計（2026-03-22 〜 2026-04-30）
- 通知モデル統合（通知種別・既読管理・配信履歴）
- 履歴/コメントのクエリ再設計（高オフセット耐性、カーソル化）
- マイグレーション方針統一（前方互換・冪等・ロールバック手順）

完了条件:
- 通知/履歴系エンドポイントの p95 が現行比で改善
- すべてのスキーマ変更にロールバック手順が付与

### Phase 3: サービス分割と移行（2026-05-01 〜 2026-06-30）
- Exchange/Notification をアプリ層で分離（use-case単位）
- 旧実装を Feature Flag 下で新実装へ段階切替
- バックグラウンド処理（matching refresh 等）をジョブ基盤へ移管

完了条件:
- 主要ユースケースで新旧切替が可能
- 障害時に即時で旧実装へ戻せる

### Phase 4: フロントエンド再構成（2026-07-01 〜 2026-08-15）
- 画面単位から機能単位へ再編（feature-first）
- API client と状態管理を統一（型生成ベース）
- 通知/提案/履歴UIを共通設計へ統合

完了条件:
- 新UI層でのE2E主要シナリオ通過
- 画面追加時の重複実装が発生しない構造になっている

### Phase 5: 収束・切替完了（2026-08-16 〜 2026-09-15）
- 旧ルート/旧サービスの段階廃止
- コスト最適化（DBインデックス、キャッシュ、ジョブ並列度）
- 運用Runbook/障害訓練の最終整備

完了条件:
- 旧実装の依存を除去
- P1/P2障害の再発率が目標以下

## 2. 優先バックログ（最初の4週間）
- 通知既読更新の単一トランザクション化（実施済み）
- 通知未読数APIの劣化耐性強化（実施済み）
- 履歴APIの高オフセット負荷対策（実施済み）
- コメント編集/削除のID列挙差分抑制（実施済み）
- `/find` の負荷制御（実施済み、次段で非同期ジョブ分離）

## 3. ガバナンス
- 週次品質ゲート: `typecheck + lint + tests + セキュリティレビュー`
- リリース判定: P1/P2ゼロ、ロールバック手順確認済み
- 変更管理: ADR（Architecture Decision Record）必須

## 4. リスクと先回り対策
- リスク: 新旧実装の二重運用で複雑化
  - 対策: 切替期限を明示し、Flag棚卸しを隔週実施
- リスク: データ移行時の欠損
  - 対策: 二重書き期間 + 差分検証ジョブ + 監査ログ
- リスク: テストの維持コスト増
  - 対策: 契約テストを中心にし、重複E2Eを削減
