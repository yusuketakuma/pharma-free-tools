# 薬歴下書き・要点整理支援 ワイヤー提案

## 結論
`pharmacy-medication-history-efficiency.html` と `medication-history-time-saving-checklist.html` は、診断・自己評価のままだと価値が止まる。  
次の一手は、**薬歴をその場で下書きできる入力/出力型ツールへ寄せる 1ページ wireframe を固定すること**。

## Board 判断
- **Board Visionary**: 高頻度・高痛み・既存資産活用のバランスが最良
- **Board User Advocate**: 薬剤師の認知負荷を下げる。入力項目が現場の言葉に近い
- **Board Operator**: まずは 1 ページの仕様固定で十分。実装前に迷いを減らせる
- **Board Auditor**: 低リスク。auth / routing / approval / trust boundary は触らない
- **Board Chair**: 今回は「改善候補」ではなく「実装できる仕様」に落とすのが採否基準

**採否: 採用**

## 目的
- SOAP の A/P、前回差分、患者説明メモを後ろ倒しにしない
- 「診断する」から「そのまま下書きを作る」へ変える
- 記録・説明・次回確認を1回の流れで出せるようにする

## 想定ユーザー
- 薬局薬剤師
- 管理薬剤師
- 薬歴入力に時間を取られている現場担当者

## 入力ワイヤー
### 1. 患者・処方の基本情報
- 年齢層
- 性別
- 既往 / 併存疾患
- 受診目的
- 処方変更の有無
- 直近の処方差分

### 2. 確認したい論点
- 副作用の有無
- 効果実感
- 服薬状況
- 飲み忘れ / 休薬
- 生活背景（仕事、食事、飲み方、介護など）
- 残薬 / 期限切れ / 受診間隔

### 3. 現場メモ
- 口頭で気になった点
- 医師に戻すべき点
- 患者へ説明したい点
- 次回までに確認したい点

## 出力ワイヤー
### 1. SOAP 下書き
- **S**: 患者訴え / 服薬状況 / 気になる変化
- **O**: 処方差分 / 参考情報 / 観察事項
- **A**: 問題点の要約 / リスク判断 / 優先課題
- **P**: 次回確認事項 / 介入案 / 医師連絡の要否

### 2. 次回確認事項
- 次回来局時に聞くこと
- 服薬継続の確認ポイント
- フォローが必要な副作用や症状

### 3. 患者説明メモ
- 1〜3文でそのまま使える説明文
- 難しい表現を避けた平易な文
- 必要なら「短く伝える版」と「丁寧版」を切り替え可能にする

### 4. 介入フラグ
- 医師確認が必要
- 緊急度高
- 様子見で可
- 追加情報待ち

## UI 構成案
1. 上段: 患者・処方の入力欄
2. 中段左: 論点チェック
3. 中段右: SOAP 下書き生成
4. 下段左: 次回確認事項
5. 下段右: 患者説明メモ
6. フッター: 保存用コピーボタン / リセット / 共有導線

## 実装上の方針
- 既存診断ロジックを捨てず、出力型に上乗せする
- まずは静的HTML内で完結する
- 外部送信や auth 連携はしない
- 保存はローカルコピー前提にする

## 成功条件
- 1回の入力で SOAP 下書きが出る
- 患者説明文がそのまま使える
- 次回確認事項が3点以内でまとまる
- 診断止まりのページより「使った感」が明確に上がる

## 非目標
- 予約システム連携
- 薬歴システムへの自動送信
- 生成AIの外部 API 連携
- Telegram 設定変更
- routing / approval / trust boundary の変更

4. 実装前にコピー文例を 3 パターン用意する
